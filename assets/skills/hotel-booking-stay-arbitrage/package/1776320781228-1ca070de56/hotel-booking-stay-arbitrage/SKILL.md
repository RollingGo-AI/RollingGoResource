# 智能连住价格预测师（StayArbitrage Advisor）

> 把同一行程拆成多段入住，找出总价最低的连住组合方案。

## 场景说明

用户在同一目的地连住多晚时，通常只搜索单一酒店并全程入住，错失了通过"分段入住不同酒店"
大幅节省费用的机会。本 Skill 分析目的地范围内多家酒店的逐日动态房价，对各种入住组合进行
优化，找出总价最低且换房成本可接受的拼接方案，并向用户清晰解释每段的推荐理由和节省金额。

## 触发示例

1. "我在成都要住 6 晚，有没有办法把住宿费压低一点？"
2. "下周去北京出差，大概住 5 天左右，怎么订比较划算？"
3. "我已经查了一家酒店全程入住要 3200 元，有没有更便宜的方案？"

## 工作流程

### Step 1：收集必要信息

**必须从用户处获取：**
- 目的地城市（如"上海静安区"、"三亚海棠湾"）
- 入住日期（精确到日）
- 总住宿晚数

**可从上下文推断：**
- 成人人数（默认 2 人）
- 可接受的最大换房次数（默认 1 次，即最多住两家酒店）

**询问话术示例：**
> "我来帮你算一下拼住方案。请问是几月几日入住、一共住几晚？是在哪个城市或区域？"

---

### Step 2：获取场景相关标签

本场景通常**不依赖标签过滤**（目标是价格优化，而非特定设施）。

若用户明确提出有设施要求（如"需要健身房"、"要带浴缸"），则先运行 `hotel-tags` 获取
精确标签字符串，再将其作为 `--preferred-tag` 加入搜索：

```bash
npx --yes --package rollinggo@latest rollinggo hotel-tags
```

---

### Step 3：执行分段搜索

**核心逻辑：** 将总住宿天数划分为 2 段（前半段 + 后半段），分别搜索同区域内性价比最高的酒店，
再计算两段总价与单一全程住宿的对比节省。

**第一轮：查询全程单住最低价（基准价）**

```bash
npx --yes --package rollinggo@latest rollinggo search-hotels \
  --origin-query "上海静安区连住6晚酒店" \
  --place "上海静安区" \
  --place-type city \
  --country-code CN \
  --check-in-date 2026-04-20 \
  --stay-nights 6 \
  --adult-count 2 \
  --size 5 \
  --format table
```

**第二轮：搜索前半段（第 1-3 晚）候选酒店**

```bash
npx --yes --package rollinggo@latest rollinggo search-hotels \
  --origin-query "上海静安区3晚酒店价格" \
  --place "上海静安区" \
  --place-type city \
  --country-code CN \
  --check-in-date 2026-04-20 \
  --stay-nights 3 \
  --adult-count 2 \
  --size 8 \
  --format table
```

**第三轮：搜索后半段（第 4-6 晚）候选酒店**

```bash
npx --yes --package rollinggo@latest rollinggo search-hotels \
  --origin-query "上海静安区后3晚酒店" \
  --place "上海静安区" \
  --place-type city \
  --country-code CN \
  --check-in-date 2026-04-23 \
  --stay-nights 3 \
  --adult-count 2 \
  --size 8 \
  --format table
```

---

### Step 4：场景专属筛选与解读

**二次筛选逻辑：**

1. 从三轮搜索结果中提取各酒店的 `hotelId`，调用 `hotel-detail` 获取精确报价：

```bash
npx --yes --package rollinggo@latest rollinggo hotel-detail \
  --hotel-id "HOTEL_ID_A" \
  --check-in-date 2026-04-20 \
  --check-out-date 2026-04-23 \
  --adult-count 2
```

```bash
npx --yes --package rollinggo@latest rollinggo hotel-detail \
  --hotel-id "HOTEL_ID_B" \
  --check-in-date 2026-04-23 \
  --check-out-date 2026-04-26 \
  --adult-count 2
```

2. **换房成本评估：**
   - 若两家酒店距离 ≤ 500 米：换房成本低，可优先推荐
   - 500 米 - 2 公里：中等成本，需提示用户
   - ＞2 公里：需在推荐中明确告知转移时间和打车费用估算

3. **节省金额计算：**
   - 节省额 = 全程单住最低价 - (前段总价 + 后段总价)
   - 节省率 ≥ 15% 才作为"值得推荐"的方案输出

**向用户解释推荐理由话术示例：**
> "第 1-3 晚住锦江之星静安寺店（¥380/晚），第 4-6 晚换到如家精选酒店（¥290/晚），
> 总价 ¥2,010，比全程住一家酒店节省 ¥680（节省 25%）。两家相距 800 米，打车约 5 分钟。"

---

### Step 5：输出结果

**输出格式：**
- 列出 1-2 个拼住方案（按节省金额从高到低）
- 每个方案包含：各段酒店名称、晚数、单价、合计、节省金额、两家距离
- 附上全程单住基准方案用于对比
- 每段酒店附 booking URL

**话术模板：**
> **推荐方案（节省最多）**
> - 第 1-3 晚：[酒店A]，¥380/晚 × 3 = ¥1,140 | [预订链接]
> - 第 4-6 晚：[酒店B]，¥290/晚 × 3 = ¥870 | [预订链接]
> - **合计：¥2,010，节省 ¥680（相比全程住 [基准酒店] ¥2,690）**
> - 换房距离：约 800 米，打车约 ¥12

---

## 本场景专属规则

1. **必须与全程单住方案做价格对比**：若无基准价，不得直接给出拼住方案，缺少对比等于没有价值。
2. **节省率低于 15% 时不推荐拼住**：换房本身有时间和精力成本，微小节省不值得推荐。
3. **换房距离超过 2 公里必须告知**：需向用户明确说明转移所需时间和估算交通费用。
4. **不得跨区域拼住**：两段酒店必须在用户指定的同一区域内，不得为追求低价推荐远离目标区域的酒店。
5. **优先选取取消政策宽松的组合**：当总价相近时，优先推荐两段均支持免费取消的方案。

---

## 无结果时的处理

**本场景专属放宽顺序：**

1. 扩大搜索范围（增加 `--distance-in-meter` 或扩大到城市级别）
2. 增加候选结果数量（`--size` 从 8 提升到 15）
3. 放宽星级限制（若用户有设置 `--star-ratings`，优先放宽）
4. 拆分方案改为"2 晚 + 4 晚"或"1 晚 + 5 晚"重新计算

**本场景不得放宽：**
- 目的地城市（用户的核心需求，不可改变）
- 总住宿晚数（行程固定，不可压缩）
- 换房次数上限（超过用户承受范围则直接回退为全程单住推荐）
